# Telegram Bot — Документация проекта

## Описание

Telegram-бот для управления заявками на обмен RUB ↔ USDT между партнёрами. Позволяет создавать, публиковать и управлять заявками в нескольких Telegram-группах одновременно, с ролевым доступом, жизненным циклом заявки, авто-истечением, флоу подтверждения сделки и рейтингом партнёров.

---

## Стек и зависимости

- **Python ≥ 3.10** (на проде — Python 3.14, Ubuntu 26.04)
- `python-telegram-bot[job-queue]==22.8` — основная библиотека для работы с Telegram API. `[job-queue]` обязателен: без него не работают TTL, автоудаление и автоподтверждение
- `aiohttp` — HTTP-сервер для healthcheck (слушает только localhost)

Зависимости зафиксированы в `requirements.txt`. (`pytz` не используется — убран.)

## Процесс создания заявки

1. **Направление** — Продам RUB / Куплю RUB
2. **Сумма** — вводится вручную
3. **Банк** — до 2 банков: СБЕР, ТИНЬКОФФ, АЛЬФА, СБП
4. **Платежи** — 1 / до 2 / до 3 платежей
5. **Курс** — Байбит средний / Курс Grinex / Свой курс
6. **Срок (TTL)** — 30 мин / 1 час / 2 часа / 3 часа / В течение дня
7. **Выбор чатов** — один или все доступные

## Жизненный цикл заявки

```
Создана → Опубликована в чатах
               ↓
         [Взята в работу] ──────────────────────────────────────┐
               ↓                                                  │
    Партнёр нажал «Выполнено»                  Мерчант закрыл сам
               ↓                                       ↓
    Мерчант получает запрос подтверждения    Сделка записана автоматически
               ↓                             Форма оценки мерчанту
    Подтвердил вручную (или авто через 15 мин)        ↓
               ↓                             Заявка удалена из всех чатов
    Мерчант ставит оценку ⭐️ (1–5)
               ↓
    Заявка удалена из всех чатов
               ↓
         [Истекла по TTL]
               ↓
    Переопубликовать / Закрыть → Автоудаление через 5 мин
```

- **Истечение**: по TTL заявка помечается как истёкшая, кнопки удаляются
- **Автоудаление**: если не переопубликована и не закрыта — через 5 минут удаляется из всех чатов
- **Переопубликация**: создаёт новую заявку с теми же параметрами
- **Принудительное закрытие**: если взятая заявка истекла и никто не среагировал — через 10 минут принудительно закрывается

---

## Флоу подтверждения сделки

1. Партнёр нажимает **«✅ Выполнено»** → мерчанту приходит сообщение с кнопками «Подтвердить» / «Оспорить»
2. Через 5 минут — **напоминание** мерчанту
3. Через 15 минут без реакции — **авто-подтверждение** сделки
4. После подтверждения — мерчанту показывается форма **оценки ⭐️ 1–5**
5. Через 15 минут если оценка не поставлена — заявка автоматически закрывается без оценки
6. **Оспорить** — заявка удаляется из всех чатов без записи сделки

**Если мерчант сам закрывает взятую заявку:**
- Сделка записывается автоматически
- Мерчанту показывается форма оценки
- Заявка немедленно удаляется из всех групповых чатов

## Система рейтинга партнёров

- Оценки хранятся в `state["ratings"][user_id]`
- Поля: `total_deals`, `total_taken`, `avg_score`, `scores[]`, `avg_completion_minutes`
- Рейтинг виден в карточке партнёра (`/partner @username`)
- Completion rate = `total_deals / total_taken * 100%`
- Среднее время выполнения в минутах

- Лимит активных заявок на мерчанта

## Важные особенности реализации

- **Все команды только в личке** — бот не отвечает на команды в группах
- **Проверка членства** — мерчанты видят только те чаты, где они состоят
- **Миграция чатов** — когда обычная группа становится супергруппой, у неё меняется chat_id. Все отправки идут через `_record_sent_message`: сохраняет сообщение в `order["messages"]` и, если `msg.chat.id` ≠ запрошенного id (миграция), чинит реестр через `_migrate_chat_id` (переносит запись чата на новый id). Это держит `state["chats"]` и `order["messages"]` в одном id-пространстве — иначе ломается дедупликация отправки (см. ниже)
- **Самолечение мигрировавших групп** — `load_state` при каждой загрузке сверяет чаты с `DEFAULT_CHATS` по имени: если чат с известным именем висит под старым id, переносит его на актуальный id из `DEFAULT_CHATS`. Так прод-`state.json` со старым id чинится автоматически при рестарте
- **Хранение chat_id** — при отправке сообщения в чат всегда сохраняется `msg.chat.id` (реальный ID из ответа Telegram), а не запрошенный ID
- **Дедупликация отправки по чатам** — флоу «отправить в оставшиеся чаты» сравнивает `available_chats` (из `state["chats"]`) с `sent_chats` (id из `order["messages"]`). Если id чата в этих двух источниках расходятся (мигрировавшая группа), кнопка чата не исчезает и заявку можно отправить дважды. Чинится тем, что и реестр, и messages держатся на одном (новом) id — см. «Миграция чатов»
- **Единая функция удаления** — `_close_order_from_chats` используется везде: удаляет из групчатов, из личного чата бота (`bot_message_id`), удаляет заявку из стейта
- **Дедупликация чатов** — при загрузке проверяется название чата, чтобы не дублировать мигрировавшие группы
- **Нельзя запускать** локальный бот и деплой одновременно (конфликт polling)
- **`bot_message_id` / `bot_chat_id`** — ID сообщения заявки в личном диалоге с ботом, хранится в заявке для корректного удаления
- **Атомарное сохранение стейта** — `save_state` пишет во временный файл + `fsync` + `os.replace`; `state.json` не бьётся при краше/ребуте во время записи
- **HTTP-сервер только на localhost** — слушает `127.0.0.1` (env `HTTP_HOST`), эндпоинт `/test/list_chats` со списком чатов наружу не торчит
- **Файл бота** ранее назывался `simple_bot (1).py` (с пробелом) — переименован в `simple_bot.py`

## Ключевые функции

| Функция | Описание |
|---------|----------|
| `_close_order_from_chats` | Удаляет заявку из всех чатов + бота, убирает из стейта |
| `_do_confirm_deal` | Ядро подтверждения сделки: запись, оценка, уведомления |
| `_auto_confirm_job` | Авто-подтверждение через 15 мин если мерчант не среагировал |
| `_auto_close_confirmed_order_job` | Авто-закрытие через 15 мин если мерчант не поставил оценку |
| `_force_close_claimed_expired_job` | Принудительное закрытие взятой истёкшей заявки через 10 мин |
| `edit_message_safe` | Редактирование с обработкой ChatMigrated и BadRequest |
| `send_message_safe` | Отправка с обработкой ChatMigrated |
| `_record_sent_message` | Сохраняет отправленное сообщение в `order["messages"]`; при миграции группы чинит реестр чатов и возвращает реальный chat_id |
| `_migrate_chat_id` | Переносит запись чата в `state["chats"]` со старого id на новый (миграция группы → супергруппа) |
| `build_keyboard` | Строит инлайн-клавиатуру в зависимости от роли и статуса заявки |
| `render_message` | Формирует текст заявки для отображения |
| `record_deal` | Записывает сделку в историю партнёра |
| `record_rating` | Записывает оценку партнёру |
